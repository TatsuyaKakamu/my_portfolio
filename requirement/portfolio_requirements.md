# 個人ポートフォリオサイト 要件定義書

作成日: 2026-04-24  
更新日: 2026-04-24  
目的: 個人ブランディング向けポートフォリオサイトの構築  
コンセプト: 信頼性のある技術者 + 継続的にアウトプットする個人  
更新方針: 外部サービス連携による全自動更新を目指す  
表示方針: 白基調・落ち着いたテーマカラー・優雅で大胆な低速アニメーション

---

## 1. 背景

現在、本人の活動実績は複数の外部サービスに分散して存在している。

- アプリ開発: GitHub の公開リポジトリ
- セキュリティ対応資格: TÜV SÜD の資格取得者ページ
- 写真投稿: Instagram
- トレーニング: Strava のトレーニングログ

本サイトでは、これらの分散した活動を一つのポートフォリオとして集約し、本人の継続的な取り組み、技術的信頼性、人柄、世界観を伝える。

---

## 2. サイトの目的

### 2.1 主目的

個人ブランディング。

採用・転職・副業案件の獲得を直接目的にするのではなく、以下を自然に伝えることを目的とする。

- 継続的にアプリ開発を行っていること
- 産業用サイバーセキュリティ領域の学習・資格取得実績があること
- 写真投稿など、技術以外の創作活動も継続していること
- トレーニングを通じて自己管理・継続性を持っていること
- 技術者としての信頼性と、個人としての世界観が両立していること

### 2.2 想定閲覧者

- 技術者コミュニティで本人を知った人
- GitHub や SNS から流入した人
- 将来的な協業相手
- 個人活動に興味を持った人
- 採用・副業関係者も副次的な閲覧者として想定する

---

## 3. 基本方針

### 3.1 表現方針

サイト全体の方向性は以下とする。

> 信頼性のある技術者 + 継続的にアウトプットする個人

技術力だけを前面に出すのではなく、以下の要素をバランスよく見せる。

- Technical: アプリ開発、GitHub、実装実績
- Trust: IEC 62443 ファウンデーション資格
- Creative: 写真投稿、視点、感性
- Discipline: トレーニング習慣、継続性、自己管理

### 3.2 公開情報の方針

公開する情報は、原則として既に公開されている情報、または本人が公開を許容した情報に限定する。

公開する:

- 実名
- 加工済み顔写真
- GitHub の公開リポジトリ
- TÜV SÜD 資格取得者ページへのリンク
- Instagram 公開アカウントへのリンクまたは一部写真
- Strava の公開可能な範囲の活動概要

公開しない:

- 会社名
- 現職の詳細
- 勤務地
- 非公開リポジトリ
- 詳細な健康情報
- 自宅周辺や生活圏が推測される位置情報
- 連絡先情報

### 3.3 更新方針

将来的に全自動更新を前提とする。

ただし、各外部サービスの API 制約、認証方式、利用規約、取得可能データの制限があるため、以下の考え方で実装する。

- 公式 API が利用できるものは公式 API を優先する
- Webhook が利用できるものはイベント駆動更新を検討する
- Webhook がないものは定期ジョブで取得する
- 非公式スクレイピングは原則避ける
- 自動取得が難しい項目は、手動管理データをフォールバックとして保持する
- 取得失敗時は直近成功データを表示する
- データの取得日時を保持し、必要に応じてサイト上に最終更新日時を表示する

## 4.2 App Development セクション

### 目的

公開アプリ・公開リポジトリを通じて、継続的な開発活動と技術的アウトプットを示す。

### 対象データ

- GitHub の公開リポジトリ
- アプリのスクリーンショット
- README の概要
- 使用技術
- 最終更新日
- GitHub リポジトリへのリンク

### 掲載内容

各プロジェクトごとに以下を表示する。

- アプリ名 / リポジトリ名
- 概要
- スクリーンショット
- 技術スタック
- 開発目的
- 工夫した点
- GitHub リンク
- デモリンクがある場合はデモリンク

### 実装方針

- GitHub REST API または GraphQL API から公開リポジトリ一覧を自動取得する
- 掲載対象は allowlist、GitHub Topics、またはローカル設定ファイルで制御する
- description、topics、language、stars、updated_at、repositoryUrl を取得する
- README の全文取得は情報量が過剰になりやすいため、概要文は手動補足を許容する
- アプリスクリーンショットは初期は手動配置し、後から OGP 画像やリリースアセット取得を検討する
- GitHub Actions、Vercel Cron、または Cloudflare Workers Cron Triggers で定期更新する

## 4.3 Security / Certification セクション

### 目的

産業用サイバーセキュリティに関する学習・資格取得実績を示し、技術者としての信頼性を補強する。

### 対象資格

- TÜV SÜD
- Certified Persons Industrial Cybersecurity Foundation (Level 1)
- According to IEC 62443
- 掲載名: Tatsuya Kakamu
- Date of Issue: 2025.6.5
- ID: M10522-25M126062

### 掲載内容

- 資格名
- 発行元
- 発行日
- 資格取得者ページへのリンク
- IEC 62443 に関する簡潔な説明
- 本人の関心領域としての産業用サイバーセキュリティ

### 注意点

- 資格を過度に誇張しない
- 「専門家」「コンサルタント」など、実態以上の表現は避ける
- 「産業用サイバーセキュリティの基礎知識を体系的に学習した証明」として扱う

### 実装方針

- TÜV SÜD 資格取得者ページへのリンクを掲載する
- 氏名・発行日・ID はローカルのデータファイルにも保持する
- 定期ジョブで資格ページを確認し、氏名または ID の掲載状態を検証する
- 掲載状態に変化があった場合は、サイトデータを更新し、必要に応じて通知する
- 外部サイトのスクレイピングは利用規約・robots.txt・アクセス負荷に注意する
- 自動取得が不適切と判断した場合は、資格情報は手動更新対象にフォールバックする

## 4.4 Creative / Photography セクション

### 目的

技術以外の創作活動として、写真投稿を控えめに紹介する。サイト全体の人柄・世界観を補強する位置づけとする。

### 対象データ

- Instagram 公開アカウント
- 代表写真
- 投稿へのリンク

### 掲載内容

- Instagram アカウントへのリンク
- 代表写真数枚
- 写真活動の簡単な説明

### 表示方針

- 技術ポートフォリオの中心には置かない
- トップページでは小さめに扱う
- 専用セクションでは写真の見栄えを重視する
- 写真は本人の世界観を伝える補助要素とする

### 実装方針

- Instagram 公開アカウントへのリンクを掲載する
- 代表写真は、公式 API または公式埋め込みで取得可能な範囲を優先する
- Instagram API の利用条件、アクセストークン更新、表示制限を確認する
- API 連携が不安定、または個人サイト用途に過剰な場合は、手動選定した写真をリポジトリ内で管理し、Instagram 投稿 URL のみ紐づける
- 写真データの無断再配布や非公式スクレイピングは避ける

## 4.5 Training セクション

### 目的

Strava のトレーニングログを通じて、継続性・自己管理能力・身体的な習慣を示す。

### 対象データ

- Strava 公開プロフィール
- Strava のアクティビティ情報
- 月間または年間の概要値

### 掲載内容

詳細ログではなく、ダッシュボード的な概要を表示する。

例:

- 今月のアクティビティ数
- 今月の総距離
- 年間アクティビティ数
- 年間総距離
- 直近の活動傾向
- 継続日数または月次サマリ

### 非掲載情報

- 詳細な心拍データ
- 住所や生活圏が推測できるルート情報
- 正確な開始地点・終了地点
- 健康状態が過度に推測される情報

### 実装方針

- Strava API を利用して、本人のアクティビティ概要を自動取得する
- OAuth 2.0 による認証を行う
- アクセストークンとリフレッシュトークンはサーバー側または CI/CD の Secret として管理する
- 取得データは集計済みの JSON として保存し、サイト表示には集計値のみを利用する
- 必要に応じて Strava Webhook を検討する
- 位置情報、地図、詳細ルート、心拍などはサイトに表示しない
- API 取得に失敗した場合は、直近成功時の集計データを表示する

## 6. データ取得・更新方針

### 6.1 基本方針

全自動更新を最終形とする。

- GitHub: API による公開リポジトリ情報の自動取得
- TÜV SÜD: 定期ジョブによる掲載状態確認。ただし規約上不適切な場合は手動管理にフォールバック
- Instagram: 公式 API または公式埋め込みで可能な範囲を自動化。困難な場合は代表写真のみ手動管理
- Strava: API による集計データ取得を自動化

### 6.2 自動更新の対象

| 対象 | 更新方式 | 表示内容 | 備考 |
|---|---|---|---|
| GitHub | REST API / GraphQL API | 公開リポジトリ、更新日、技術情報 | allowlist で掲載対象を制御 |
| TÜV SÜD | 定期ページ確認 | 掲載状態、資格情報 | 規約・robots.txt を確認 |
| Instagram | 公式 API / Embed | 代表投稿またはリンク | 非公式スクレイピングは避ける |
| Strava | Strava API / Webhook / Cron | 月次・年次サマリ | 位置情報と健康詳細は除外 |

### 6.3 更新頻度の目安

| 対象 | 更新頻度 | 理由 |
|---|---|---|
| GitHub | 1日1回、またはビルド時 | 更新頻度が比較的高い |
| TÜV SÜD | 週1回程度 | 頻繁な変化は想定しにくい |
| Instagram | 1日1回、または手動同期 | API制約を考慮 |
| Strava | 1日1回、またはアクティビティ作成時 | 継続性ダッシュボード用 |

### 6.4 データ処理フロー

```text
External Services
  ├── GitHub API
  ├── Instagram API / Embed
  ├── Strava API / Webhook
  └── TÜV SÜD page check
        ↓
Data Fetch Layer
        ↓
Normalize / Privacy Filter
        ↓
Generated JSON / Markdown
        ↓
Static Site Build
        ↓
Portfolio Site
```

### 6.5 失敗時の挙動

- API取得に失敗してもサイト表示は壊さない
- 直近成功時のキャッシュデータを使う
- データ更新日時を保持する
- 連続失敗時はビルドログまたは通知で検知できるようにする
- エラー内容を公開サイト上に直接表示しない

## 7. 技術構成案

### 7.1 推奨構成

将来の全自動更新を考慮し、以下の構成を第一候補とする。

- フレームワーク: Astro または Next.js
- ホスティング: Cloudflare Pages / Vercel / GitHub Pages
- データ管理: Markdown + JSON / YAML
- 自動更新: GitHub Actions、Vercel Cron、Cloudflare Workers Cron Triggers
- 画像管理: リポジトリ内 assets または外部画像 CDN
- Secret 管理: GitHub Actions Secrets、Vercel Environment Variables、Cloudflare Secrets

### 7.2 Astro 案

静的サイト中心で、表示速度・保守性を重視する場合に適する。

利点:

- 静的サイトとして軽量
- Markdown との相性が良い
- ブログ・記事追加に向いている
- GitHub Pages / Cloudflare Pages と相性が良い
- GitHub Actions による定期データ生成と組み合わせやすい

懸念:

- Strava Webhook などイベント駆動更新を行う場合、別途 API 受け口が必要

### 7.3 Next.js 案

API連携や将来的な拡張性を重視する場合に適する。

利点:

- API連携・定期取得との相性が良い
- Vercel Cron との組み合わせが容易
- 動的ページや将来の管理画面にも拡張しやすい
- Strava Webhook の受け口を実装しやすい

懸念:

- 単純なポートフォリオとしてはやや重い
- 運用設計が複雑になりやすい

### 7.4 初期推奨

初期は Astro + GitHub Actions を推奨する。

理由:

- 個人ブランドサイトとして高速・軽量
- Markdown / JSON による管理がしやすい
- MVPを短期間で構築しやすい
- GitHub Actions で外部サービスから定期取得し、静的サイトに反映できる

ただし、Strava Webhook や将来的な管理画面を重視する場合は、Next.js + Vercel も有力候補とする。

## 8. デザイン要件

### 8.1 トーン

- 清潔感
- 技術者らしい信頼感
- 過度に企業サイト風にしない
- 写真・トレーニング要素により個人性を出す
- 余白を広く取り、情報を詰め込みすぎない
- 白基調で、軽さと静けさを感じる画面にする

### 8.2 背景

- 背景は白基調とする
- 完全な純白だけでなく、必要に応じてごく薄いグレーやオフホワイトを使用してよい
- セクション間の区切りは強い罫線ではなく、余白、薄い背景差、カードの影で表現する
- 暗色背景を主役にしない

### 8.3 テーマカラー

各活動領域にテーマカラーを設定する。

| 領域 | テーマカラー | 印象 | 使用箇所 |
|---|---|---|---|
| App Development | 緑 | 開発、成長、継続的改善 | セクション見出し、アイコン、カードアクセント |
| Security / Certification | 青 | 信頼、冷静さ、堅牢性 | 資格カード、検証リンク、バッジ |
| Creative / Photography | 紫 | 感性、創造性、個性 | 写真セクション、ギャラリーアクセント |
| Training | オレンジ | 活力、習慣、前進 | トレーニングダッシュボード、数値カード |

### 8.4 色の扱い

- いずれのテーマカラーも過度に強調しない
- 彩度は抑え、落ち着きのある色合いにする
- 背景全面に強い色を敷かない
- テーマカラーは以下に限定して使う
  - 小さなアクセントライン
  - アイコン
  - セクションラベル
  - カードのホバー時アクセント
  - ダッシュボード数値の補助色
- 本文、説明文、主要UIは黒・グレー系を中心にする

### 8.5 視覚表現

- 白・黒・グレーを基調
- セクションごとに落ち着いたテーマカラーを持たせる
- 写真を使うが、主役は情報設計
- セクションごとにカードUIを使用
- GitHub や資格情報は信頼性が伝わる控えめなデザイン
- 写真セクションでは余白と画像比率を重視する

### 8.6 レスポンシブ対応

本サイトはレスポンシブ対応を必須とする。

対応対象:

- スマートフォン
- タブレット
- デスクトップ
- ワイドディスプレイ

方針:

- モバイルファーストで設計する
- スマートフォンでは1カラム構成を基本とする
- タブレット以上では2カラムまたはカードグリッドを使用する
- デスクトップでは余白を広く取り、情報密度を上げすぎない
- ナビゲーションは画面幅に応じて切り替える
- 画像、カード、ダッシュボードは横幅に応じて自然に再配置する
- タップ対象は十分なサイズを確保する
- 横スクロールが発生しないようにする

### 8.7 ブレークポイント目安

| 幅 | 想定デバイス | 表示方針 |
|---|---|---|
| 〜639px | スマートフォン | 1カラム、縦積み、簡潔な表示 |
| 640〜1023px | タブレット | 2カラム中心、カード表示 |
| 1024px〜 | デスクトップ | グリッド表示、余白を広めに確保 |
| 1440px〜 | ワイド画面 | 最大幅を制限し、中央寄せ |

### 8.8 アニメーション / モーション

動作アニメーションは、優雅に大胆に、速度は遅めにする。

方針:

- 小刻みでせわしないアニメーションは避ける
- ページ全体の印象として、ゆっくり、静かに、余韻のある動きを使う
- セクションの出現、カードのホバー、画像の切り替えなどに限定して使用する
- 情報理解を妨げる過剰な演出は避ける
- ユーザー操作への応答は遅くしすぎない
- `prefers-reduced-motion` に対応し、動きを減らす設定のユーザーにはアニメーションを抑制する

### 8.9 アニメーション例

- ファーストビューのテキストをゆっくりフェードインする
- セクションが画面内に入ったとき、少し大きめの移動量でゆっくり出現させる
- カードホバー時に、控えめに浮き上がる
- 写真は急に切り替えず、ゆっくりクロスフェードする
- トレーニング数値は派手にカウントアップせず、控えめに表示する

### 8.10 アニメーション速度目安

| 対象 | 速度目安 | 備考 |
|---|---|---|
| ページ初期表示 | 800〜1200ms | 優雅さを出す |
| セクション出現 | 700〜1000ms | 大きめの移動量でもよい |
| カードホバー | 250〜400ms | 操作感を損なわない範囲 |
| 写真切り替え | 900〜1400ms | ゆっくりした余韻を出す |
| ナビゲーション | 300〜500ms | 遅すぎない |

### 8.11 避ける表現

- 過度な自己演出
- 企業名や現職情報が推測される表現
- 資格を実力以上に見せる表現
- トレーニングログの過剰な詳細表示
- SNS投稿の全面的な埋め込み
- 強い原色の多用
- 背景全面に濃いテーマカラーを敷く表現
- 高速でちらつくアニメーション
- 意味のないパララックスや過剰なスクロール演出

---

## 9. 非機能要件

### 9.1 パフォーマンス

- トップページは軽量にする
- 外部埋め込みを多用しない
- 画像は圧縮・リサイズする
- Lighthouse の Performance を意識する
- アニメーションは GPU 負荷が高くなりすぎないようにする
- 大量の画像は遅延読み込みする

### 9.2 アクセシビリティ

- コントラスト比を確保する
- テーマカラーだけで意味を伝えない
- キーボード操作に対応する
- 画像には適切な alt テキストを設定する
- `prefers-reduced-motion` に対応する
- スクリーンリーダーで主要構造が理解できるようにする

### 9.3 セキュリティ・プライバシー

- APIキーやアクセストークンをクライアントに露出しない
- 位置情報を含むデータは公開前に加工する
- 顔写真は加工済みのものを使う
- 会社名・勤務地・業務詳細を出さない
- 外部サービス連携時は規約を確認する
- Strava の詳細ルート、心拍、開始地点、終了地点は公開しない

### 9.4 保守性

- コンテンツはコードから分離する
- 各活動データは JSON / YAML / Markdown で管理する
- 自動取得できない項目は手動更新しやすい形式にする
- 将来の自動化に備えてデータ構造を先に定義する
- デザイン定義は CSS Variables またはデザイントークンとして管理する

## 13. リスクと対応

| リスク | 内容 | 対応 |
|---|---|---|
| 外部API制限 | Instagram / Strava の取得制限 | 公式APIを優先し、キャッシュと手動フォールバックを用意 |
| スクレイピング制約 | TÜV SÜD ページの自動取得が不適切な可能性 | 手動リンクとローカルデータを保持 |
| プライバシー | 本名と趣味・トレーニングが結びつく | 公開範囲を限定 |
| 位置情報漏洩 | Strava のルート情報 | 詳細ログ・地図・開始終了地点を非掲載 |
| 保守負荷 | 自動化が増えるほど壊れやすい | API別に責務を分離し、キャッシュを持つ |
| 情報過多 | 技術・写真・運動が散漫になる | コンセプトを統一 |
| アニメーション過多 | 表示が重くなる、読みづらくなる | 使用箇所を限定し、reduced motionに対応 |
| 色の主張過多 | 活動領域ごとの色が散漫に見える | 彩度を抑え、アクセント用途に限定 |
| レスポンシブ崩れ | スマートフォンで見づらくなる | モバイルファーストで設計・確認 |

## 14. 成功条件

MVP時点の成功条件は以下とする。

- サイトを見た人が、本人の活動領域を30秒以内に理解できる
- アプリ開発の実績が GitHub とスクリーンショットで確認できる
- IEC 62443 ファウンデーション資格が外部リンクで確認できる
- 写真とトレーニングが人柄・継続性として自然に伝わる
- 会社名や私生活の詳細を出さずに個人ブランドを表現できる
- GitHub / Strava を中心に自動更新できるデータ構造になっている
- スマートフォン、タブレット、デスクトップで破綻なく表示できる
- 白基調で落ち着きがあり、各活動領域のテーマカラーが自然に機能している
- アニメーションが優雅で大胆だが、情報理解を妨げない
